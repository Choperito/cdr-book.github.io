6.1 Introducción

6.1.1 ¿Por qué informes reproducibles?

El resultado final de nuestro proyecto de análisis de datos terminará comunicándose a distintos niveles, tanto aguas arriba como aguas abajo. Esta comunicación es “la última milla” del flujo de análisis que se esquematizaba en la figura ??. Vamos a llamar genéricamente “informes” a cualquiera de estos resultados que se pueden producir en distintos formatos de destino. Estos informes estarán compuestos de múltiples elementos como texto, gráficos, resultados numéricos, tablas, etc. Además, es posible que haya que generarlos en distintos formatos como la web, documentos imprimibles o presentaciones. Finalmente, con una alta probabilidad varias personas intervendrán en el proceso, y la trazabilidad (reproducibilidad) del análisis mejorará el proyecto de forma global. Esta situación se esquematiza en la figura 6.1.

Figura 6.1: Esquema del proceso de información de análisis de datos

La forma de abordar el problema es típicamente con un enfoque corta-pega, en el que primero se realiza todo el análisis de datos con el software estadístico y después se utilizan los resultados del análisis como base de un informe escrito, posiblemente con algunas iteraciones del proceso si el proyecto tiene cierta envergadura. El software estadístico suele incluir formas de generar resultados listos para integrar en un informe, pero siempre bajo este paradigma de incluir el resultado a posteriori (leisch2002sweave?).

Este enfoque provoca inconsistencias (por ejemplo entre unos grupos y otros, entre diferentes analistas, etc.), errores, contenidos desactualizados o no reproducibles, especialmente en la ejecución de software, simulaciones, etc. Además, cada vez que hay que hacer un cambio, hay que hacerlo en muchos sitios, con la consiguiente pérdida de tiempo y posibles errores.

El enfoque de la investigación reproducible6 supera muchos de los obstáculos a la hora de preparar informes de análisis de datos. El objetivo es unir instrucciones para análisis de datos con datos experimentales de forma que los resultados se puedan volver a obtener, entender mejor y verificar.

Un concepto muy relacionado que utilizamos en R es la programación “literaria”7, mediante la cual combinamos un lenguaje de programación como R con documentación de todo tipo (por ejemplo comentarios en el código).

Con el enfoque de la investigación reproducible, lo que hacemos es darle la vuelta al enfoque corta-pega, de forma que se escribe el informe a la vez que se realiza el análisis, incrustando el código dentro del propio informe. Obviamente es necesario un sistema que consolide el informe con los resultados del código, y esto es lo que nos permite R y RStudio mediante archivos R Markdown.

El flujo de trabajo sería el siguiente (véase la figura 6.2): los contenidos se encuentran en ficheros de código. Estos ficheros fuente, se compilan y producen los materiales en los formatos necesarios. Los cambios se hacen una vez, y todos los materiales son actualizados adecuadamente.

Figura 6.2: Flujo de trabajo reproducible

Las ventajas de utilizar un enfoque reproducible se pueden resumir en:

  • Si el mismo analista tiene que volver al análisis en el futuro, los resultados se pueden volver a obtener de nuevo fácil y comprensiblemente.

  • En el caso de que en el proyecto participen más analistas, toda la explicación está a mano.

  • Cualquier cambio en un punto del análisis (por ejemplo, añadir una variable a un modelo) se puede realizar de una sola vez y los cambios en los resultados y gráficos se actualizarán automáticamente.

  • Los resultados se pueden verificar por terceros en caso necesario. Un caso paradigmático fue el escándalo de los ensayos de cáncer en Duke en 20118. No obstante es un tema que cada vez se demanda más en otros campos fuera de la investigación clínica (por ejemplo en publicaciones de cualquier tipo).

A continuación nos centraremos en el enfoque reproducible. Para el otro enfoque, simplemente copie los resultados de la consola y los gráficos de la pestaña Plots del panel inferior derecho en su editor de textos favorito.

6.1.2 Markdown, R Markdown y RStudio

Markdown es un tipo de ficheros de texto pensado para que se pueda leer bajo cualquier circunstancia, con una sintaxis muy simple que permite leerlo directamente por las personas, o ser convertido por un ordenador en otro formato más elaborado, como por ejemplo HTML (página web) o Microsoft Word. En RStudio, se pueden crear ficheros R Markdown utilizando esta sintaxis para las explicaciones de nuestro análisis, e incluir dentro “trozos” (chunks) de código de forma que al generar el informe, el resultado de ese código queda incluido en el documento de salida. Así, si una vez terminado el informe se nos ha olvidado, por ejemplo, incluir un gráfico, sólo tenemos que añadir las líneas de código que lo crean y volver a generar el informe.

Para poder utilizar las capacidades de R Markdown, necesitamos tener instalado el paquete knitr (xie2017dynamic?), que utiliza también otros paquetes como rmarkdown. Aunque knitr no forma parte del tidyverse, sí es un enfoque moderno de R que vino a hacer más fácil la generación de documentos que se hacía en R base con la función Sweave (leisch2002sweave?).

Para crear un nuevo documento R Markdown en RStudio, seleccionamos R Markdown … (1) en el icono de nuevo archivo (2) de la barra de herramientas o en el menú File (3), véase la figura 6.3. Entonces se abre el cuadro de diálogo New R Markdown de la figura 6.4. Vemos que hay varios tipos de archivos que se pueden crear (1), que producirán formatos diferentes: Document (documento), Presentation (Presentación de diapositivas), Shiny (Aplicación web interactiva) y From Template (usando plantilla). De momento vamos a centrarnos en los documentos. Podemos crear un archivo R Markdown vacío (2) si no queremos que nos cree la estructura. Para que se cree con una estructura mínima, se necesita un título del documento (3) y un autor (4), que después se podrán cambiar. También seleccionaremos un formato de salida por defecto (5), que puede ser HTML (para ver en el navegador), PDF, o Word. Esto también se podrá cambiar después, por lo que la forma más eficiente de trabajar es empezar con HTML, cuya previsualización es más rápida, y cuando tengamos el resultado final generar el archivo en el formato deseado.

Figura 6.3: Crear nuevo documento R Markdown

Figura 6.4: Cuadro de diálogo nuevo archivo R Markdown

Una vez hacemos clic en el botón OK de la ventana New R Markdown, figura 6.4 (6), se abre en el editor de RStudio un archivo R Markdown con una estructura básica a modo de plantilla, véase la figura 6.5. Los elementos principales de un archivo R Markdown aparecen en esta plantilla:

  • Encabezado YAML (1): Constituyen la configuración del documento, y controlan sobre todo las opciones de salida, cómo será el resultado. Este encabezado se encuentra entre dos líneas con tres guiones (---), donde se expresan las opciones como opcion: valor, y estos valores además se pueden anidar.

  • Texto formateado (2): Con una sintaxis muy sencilla, podemos dar formato al texto, como negritas, listas, etc.

  • Fragmentos de código (chunks) (3): Al generar el documento, se ejecutará el código dentro de estos fragmentos, y en el documento resultante se mostrará el resultado. En cada fragmento de código al situar el ratón sobre ellos aparecen tres botones que nos sirven para configurar las opciones del chunk, ejecutar todos los chunks anteriores y ejecutar el chunk actual.

La barra de herramientas del editor ofrece nuevas opciones:

  • El botón Knit (tejer) (4) “renderiza” el documento R Markdown produciendo el archivo de salida configurado. Se puede desplegar un menú para cambiar el formato de salida y otras opciones.

  • El botón de opciones (5) permite realizar algunos ajustes generales de entrada y salida.

  • El botón Insert (6) nos permite insertar un nuevo fragmento de código. Se pueden insertar fragmentos de otros lenguajes diferentes a R, pero en este libro no lo haremos.

  • Con las flechas de navegación (7) podemos movernos entre los chunks del documento. También podemos usar el selector de esquema (8) para ir a un fragmento de código o apartado concreto del documento.

  • Desde el menú Run (9) se puede ejecutar el código de los distintos chunks.

  • El menú Publish nos permitiría publicar el documento en algún servicio como RPubs (10)

  • El botón Outline (11) muestra un esquema para navegar por el documento (12), donde aparecerán los encabezados formateados con Markdown.

6.2 Documentos R Markdown

6.2.1 Encabezado YAML y configuración

Las opciones de configuración del documento se establecen en este encabezamiento. Generalmente no tendremos que preocuparnos de este encabezamiento, ya que se crea automáticamente al crear el documento, y se va actualizando a medida que cambiamos opciones, como veremos a continuación. No obstante está bien conocer su funcionamiento por si modificamos alguna opción (por ejemplo el título del documento), o necesitamos añadir opciones que no están en los menús de RStudio.

Al crear el documento con la plantilla, se nos ha creado el siguiente encabezado:

---
title: "Untitled"
output: html_document
---

Si hubiéramos introducido texto en la casilla “Author” (ver figura 6.4) al crear el documento, tendríamos dos líneas más, como en la figura 6.6:

---
title: "Untitled"
author: "Emilio L. Cano"
date: "6/29/2020"
output: html_document
---

Los elementos title, author y date (título, autor y fecha) son cadenas de texto que aparecerán al principio del documento de salida. Estos valores es posible que queramos modificarlos a posteriori, pero debemos cuidar que estén entre comillas para evitar posibles errores. El elemento output indica el formato de salida. También podemos modificarlo nosotros, pero no hace falta porque si queremos un formato de destino distinto basta ir al menú Knit y elegirlo. Por ejemplo, si elegimos Word, se añadirá como formato de salida word_document.

6.2.2 Formateado de texto

Incluir títulos, énfasis en el texto y listas es muy sencillo, y a menudo no necesitamos nada más para realizar un informe. En el sencillo informe que se crea con la plantilla ya vemos algunas opciones:

  • Los encabezados se crean poniendo al principio de la línea tantos símbolos almohadilla (##) como nivel de título queramos (dos almohadilla, apartado, tres almohadillas, subapartado, etc.) Una almohadilla sería para el título del informe, si no se especificara en el encabezado.

  • Para poner texto en negrita, lo rodeamos con dos asteriscos a cada lado: **negrita**.

  • Para poner texto en formato monoespaciado, tipo código, lo rodeamos de acentos graves (backticks)

  • Los enlaces se crean automáticamente si tenemos una url completa.

Existen otras opciones sencillas, que podemos ver en la Markdown Quick Reference del menú Help (figura 6.7):

  • Cursiva rodeando el texto con un solo asterisco a cada lado (o guión bajo)

  • Listas poniendo al principio de la línea un asterisco, guión o signo más

  • Listas ordenadas poniendo un número y un punto al principio de la línea.

  • Enlaces con texto + link como: [texto](url)

  • Imágenes de cualquier tipo como: ![](ruta_a_la_imagen)

  • Superíndices y subíndices

  • Ecuaciones en formato LaTeX

  • Tablas

  • Saltos de línea

  • Saltos de página

Usar plantilla de Word distinta

RStudio genera los informes usando una plantilla predeterminada. Para personalizar archivos HTML, podemos incluir hojas de estilo CSS. 6.2.3 Inclusión de código

Se pueden crear archivos R Markdown sin incluir nada de código, simplemente para crear documentos editables fácilmente. Sin embargo, la verdadera potencia de R Markdown es la posibilidad de incluir código de R (y también de otros lenguajes) en los documentos. Como ya se avazó, el código se incluye, principalmente, en forma de chunks o bloques de código.

Un chunk consta de unos marcadores de inicio y final del chunk, entre los cuales se insertan expresiones de R que se ejecutarán al generar el documento de salida. El marcador de inicio son tres símbolos de acento grave (backticks, ` ) seguidos de unas llaves con la letra r dentro. Opcionalmente, en el marcador de inicio se puede añadir un identificador del chunk y algunas opciones separadas por comas en forma de pares opcion=valor. El marcador de cierre del chunk son de nuevo tres acentos graves, sin más.

El chunk que se muestra a continuación tiene como identificador “ejemplo”, y como opciones “echo=FALSE, fig.align=‘center’”, lo que indica que el código no se mostrará en el informe final y que el gráfico producido se alineará en el centro del texto.

 ```{r ejemplo, echo=FALSE, fig.align='center'}
 plot(cars)
 ```

En todo caso, no tenemos que escribir los marcadores de inicio y final, ya que tenemos el atajo de teclado CTRL+ALT+I que lo hace por nosotros, o, alternativamente, el icono Insert de la barra de herramientas del editor (ver figura 6.5 (6)) . Una vez tenemos el cursor dentro de un chunk, podemos ejecutar una expresión como si estuviéramos en un script (CTRL+ENTER), o el chunk completo (MAYUS+CTRL+ENTER).

En la plantilla que se crea al crear un nuevo archivo R Markdown, podemos ver que se crea un bloque con el identificador setup. Este bloque es conveniente por ejemplo para cargar los paquetes que se van a configurar u opciones para aplicar en todo el documento.

A veces es necesario incluir algún resultado de R en medio del texto y no como un bloque. En esos casos podemos insertar un bloque en línea poniendo, dentro de dos acentos graves, la letra r como primer carácter, y después una expresión de R que se pueda “imprimir” como cadena de texto:

`r expresion_de_R `

Informes parametrizables

En la figura 6.8 vimos que había una opción para renderizar con parámetros. Esta opción es muy útil para informes automatizados que pueden cambiar dependiendo de algún valor, por ejemplo del fichero de datos, la fecha, o cualquier otro valor. Estos parámetros se crean como elementos del encabezado de la forma:

params:
  parametro: valor

que después se pueden usar en los chunks de código como params$parametro. La verdadera potencia de esta característica es cuando renderizamos el documento desde un script en el que los parámetros son resultados de algún tipo de operación en los datos (por ejemplo, un informe de análisis de inventario solo de una tienda donde se han producido roturas de stock el día x). En vez de utilizar el botóh Knit, en estos casos usamos la función render, una de cuyas opciones es params, donde le pasaríamos los parámetros en forma de lista cuyos elementos tienen el nombre de los parámetros.

6.2.4 Opciones de los bloques de código (chunks)

Al renderizar un informe que contiene chunks sin configurar ninguna opción, por defecto mostrará en el informe el código de entrada y las salidas (textos y gráficos) así como todos los mensajes que se pueden producir. Por defecto, si se produce un error el proceso se detiene y no se genera el archivo de destino. Este comportamiento, y otras muchas opciones, se pueden configurar en el encabezado del chunk. Volvamos al informe creado con la plantilla, figura 6.13. Fijémonos primero en el segundo chunk (1), que no tiene ninguna opción configurada, solamente una etiqueta, que nos sirve entre otras cosas para navegar por los chunks con el selector de la parte inferior (2). Por defecto, se imprimen en el informe tanto las entradas (código) como las salidas. Sin embargo en el tercer bloque (3) se ha configurado la opción echo=FALSE, lo que hace que no se muestre el código, pero sí el resultado, en este caso un gráfico. El primer bloque de código tenía configurada la opción include=FALSE (4), lo que indica que no queremos que el informe incluya ninguna entrada ni salida de ese chunk. En este bloque lo que se ha hecho es configurar el documento para que, por defecto, los chunks sí muestren el código (que es lo que ha hecho el segundo chunk al no tener configurada otra cosa). Con expresiones como la del primer chunk (5) se pueden configurar las opciones por defecto de todos los chunks del documento:

knitr::opts_chunk$set(opcion=valor)

aunque después se puede cambiar en cualquier chunk añadiendo al encabezado opcion=otro_valor. En la parte superior derecha de los chunks tenemos también unos botones (6) con los que podemos ejecutar el chunk entero (equivalente a pulsar CTRL+MAYÚS+ENTER), o todos los chunks anteriores. También tenemos un icono de opciones de chunk que abre un pequeño cuadro para las configuraciones más habituales: lo que queremos mostrar (1), si queremos o no mostrar warnings (2), o mensajes (3), o el ancho y/o alto de las figuras creadas en el chunk (4). A medida que vamos cambiando cosas, el encabezado del chunk se va actualizando.

6.3 Otros formatos

El formato R Markdown es solo uno de los que se pueden utilizar para aplicar la reproducibilidad que motiva esta unidad. En R base se pueden crear archivos Sweave, con sintaxis $ $ para la narrativa y bloques de código R. También es posible crear archivos R HTML, con la narrativa en HTML. Los identificadores de los bloques son ligeramente distintos, así como las opciones disponibles, aunque la filosofía es la misma. R Markdown ha ido desplazando estos otros formatos al ser más versátil (puede generar cualquiera de los otros formatos, y además otros como Word).

En cuanto a los formatos de salida, hay una cantidad de opciones muy interesante que queda fuera de los objetivos de este libro. Se relacionan a continuación algunos de ellos, se puede consultar el libro de (xie2019r?) para ver detalles.

  • Notebooks: Es un tipo especial de salida HTML, más indicado cuando se quieren ir probando cosas y viendo el resultado.

  • Presentaciones: Es posible crear presentaciones PowerPoint y usar una plantilla, como vimos con los documentos de Word. Se utiliza una sintaxis muy sencilla y se puede incluir código y resultados igual que en un informe. Otros formatos de presentaciones son ioslides, slidify o xaringan (HTML) y beamer (LaTeX).

  • Tableros (dashboards): Pueden ser estáticos, usando el paquete flexdashboard, útiles para comunicar resultados en un par de pantallazos.

  • Shiny: Aplicaciones web interactivas que responden a inputs del usuario (reactive)

  • Websites: Websites sencillos con páginas enlazadas en el mismo directorio.

  • Blog: Con el paquete blogdown se pueden generar sitios web completos al estilo de un blog con páginas.

  • Libros: Con el paquete bookdown se pueden crear libros en varios formatos. El material de este libro está creado de esta manera.

  • Tutoriales: Es una nueva característica de RStudio 1.3, se pueden crear documentos interactivos con preguntas y navegación.

  • Tufte Handouts: Un tipo especial de documento con anotaciones al margen que comunica muy bien.

Por último, añadir que se pueden incluir en los informes tablas formateadas de calidad. Para ello, se debe utilizar alguna función que formatee la tabla de acuerdo al formato de salida (HTML, PDF, Word), y configurar la opción results del chunk como 'asis'. Muchas de estas funciones preparan automáticamente el formato según el formato de salida que se está generando. Por ejemplo, el siguiente chunk generaría una tabla en cualquiera de los formatos de salida usando la función kable del paquete nitr:

 ```{r ejemplo, echo=FALSE, results='asis'}
 knitr::kable(head(cars))
 ```

Hay otros paquetes con multitud de opciones de formato y presentación para las tablas como xtable, flextable o kableExtra

RESUMEN

  • En la comunicación de resultados, es esencial que sigamos un enfoque reproducible.
  • R Markdown es el formato más versátil para crear informes reproducibles que permitan una trazabilidad de los análisis.
  • RStudio permite trabajar eficientemente con R Markdown a través de ayuda y opciones.
  • La narrativa del informe se escribe en Markdown, con una sintaxis extremadamente sencilla.
  • El código se puede incluir en forma de bloques (chunks) o en línea.
  • El encabezado del chunk puede incluir opciones de configuración para personalizar la forma en que se ejecutará y mostrará en el informe.
  • Los informes R Markdown se pueden generar en formatos HTML, PDF y Word.
  • Hay otros formatos más elaborados que merece la pena explorar.
